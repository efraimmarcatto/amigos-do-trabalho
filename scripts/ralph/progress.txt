## Codebase Patterns
- Dynamic Button nodes added as children of Node2D don't reliably receive GUI input — handle clicks directly in `_input`/`_handle_edit_input` using `btn.get_global_rect().has_point(event.position)` and call the action function explicitly
- Use `GIT_CONFIG_GLOBAL=/dev/null` with `-c user.name=... -c user.email=...` to work around broken gpg.format in .gitconfig
- UI panels (shop, coin HUD, etc.) are positioned programmatically in `setup()` using `floor_y` and `DisplayServer.screen_get_size()`
- All interactive UI must be included in the passthrough polygon via `_update_passthrough()` in main.gd
- In edit/placement mode, passthrough is disabled (empty polygon) so window captures all input
- Godot Buttons fire `pressed` on mouse RELEASE — when consuming input in `_input`/`_handle_edit_input`, must let through both press AND release for UI elements
- For shop/inventory grid icons: use `EXPAND_IGNORE_SIZE` + explicit `size` + `clip_contents` on parent to keep TextureRects within bounds
- Prefer tree order over z_index for rendering layers — `move_child(node, 0)` inserts at back; z_index affects ALL CanvasItem siblings (Node2D AND Control)

## 2026-03-21 - US-002
- Fixed save button click in layout edit mode
- Root cause: `_handle_edit_input` consumed ALL mouse release events (line 1021) via `set_input_as_handled()`, preventing Godot Buttons from firing their `pressed` signal (which triggers on release by default)
- Fix: Added `_is_click_on_menu_or_panels(event.position)` check before consuming release events, returning early so the event reaches the Button's GUI handler
- Files changed: scripts/main.gd
- **Learnings for future iterations:**
  - Godot Button `pressed` signal fires on mouse RELEASE (ACTION_MODE_BUTTON_RELEASE is default) — both press AND release events must reach the Button for it to work
  - When `_handle_edit_input` consumes input, it must let through BOTH press and release for menu/panel clicks, not just press
  - No typecheck tooling available for this Godot project — verify GDScript syntax manually

## 2026-03-21 - US-003
- Fixed shop panel image sizing and alignment
- Root cause: `EXPAND_FIT_WIDTH_PROPORTIONAL` allowed TextureRect to grow beyond 48x48 target when source textures were larger; `PRESET_CENTER` anchors didn't constrain size
- Fix: Changed `expand_mode` to `EXPAND_IGNORE_SIZE` so TextureRect respects only `custom_minimum_size`/`size`, set explicit `size` to ICON_SIZE, and enabled `clip_contents` on the parent Button to prevent any overflow
- Files changed: scripts/shop.gd
- **Learnings for future iterations:**
  - `EXPAND_FIT_WIDTH_PROPORTIONAL` expands the TextureRect to fit the parent width, not the target size — use `EXPAND_IGNORE_SIZE` when you want the TextureRect constrained to its own size properties
  - Always set `clip_contents = true` on parent containers when child textures may be larger than the target area
  - `STRETCH_KEEP_ASPECT_CENTERED` handles the centering/aspect ratio within the given size — no need for additional anchor tricks
  - Manual browser verification needed (no browser testing tools available)

## 2026-03-21 - US-004
- Fixed pet rendering order: pet now renders in front of furniture but behind all UI panels
- Root cause: `pet_sprite.z_index = 1` made pet render on top of ALL z_index=0 nodes, including UI Control panels. `FurnitureContainer` was added dynamically at the end of the tree (after UI panels) with z_index=0
- Fix: Removed all z_index overrides; instead insert FurnitureContainer as first child (`move_child(_furniture_container, 0)`) so tree order alone gives correct layering: FurnitureContainer → PetSprite → UI Controls
- Files changed: scripts/main.gd
- **Learnings for future iterations:**
  - In Godot 4, z_index affects ALL CanvasItem siblings (both Node2D and Control) — setting z_index=1 on a Node2D will make it render on top of sibling Control nodes with z_index=0
  - Prefer tree order over z_index for rendering layers — insert dynamic nodes at the correct position with `move_child(node, index)`
  - The `add_child()` call adds to the end; use `move_child()` immediately after to position correctly
  - Manual browser verification needed (no browser testing tools available)
